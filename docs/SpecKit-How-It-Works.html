SpecKit for Salesforce · Spec-Driven Development in Cursor
A Salesforce delivery engine on a Spec-Driven-Development framework that turns a plain-English input (a meeting transcript, notes or idea) into User Stories → Wireframes → Technical Plan/Design → Technical Tasks → Test Cases → working Salesforce code, repeatably. The AI thinking runs on Cursor + its language model; the engine adds the structure, guardrails and tracking.

2What it saves & solves — and to what extent

An accelerator, not an autopilot — four things leadership gets, with a human at every gate.

⏱️ Faster time-to-value. Stories, wireframes, plan, tasks, tests & a code scaffold in hours, not weeks — the analysis-to-build journey is compressed.
🛡️ Lower delivery risk. Ambiguity and design/security gaps are caught before code — in design, not in UAT or production.
⚙️ Consistency & governance built-in. Every feature follows the same guardrails — quality no longer depends on who builds it.
📊 Transparency & continuity. Live story → task → test traceability and the “why” retained in version-controlled specs — it survives people changes.

⚠️ Caveats — read before you trust the output

  • Manual validation is still required. Every artefact — spec, wireframe, plan, tasks, tests and code — must be reviewed and verified by a human before it is accepted or deployed. The engine drafts; people decide.
  • Output accuracy tracks input quality. The richer and more specific the starting input — a detailed transcript, the data model, sample documents, field lists, business rules — the more accurate and usable the output. Thin or vague input produces generic, assumption-heavy results.
  • The AI will fill gaps with assumptions. Where context is missing it makes its best guess; those assumptions need a human or SME to confirm or correct — which is exactly what the Clarify gate is for.
  • It accelerates experts, it does not replace them. Architects, BAs & SMEs still own the design decisions, the edge cases and the final business sign-off.
  • Always validate in a sandbox. Generated code is deployed and tested in a non-production org first — never straight to production.
Rule of thumb: the engine is only as good as what you feed it — better input → better output. Treat it as a fast, disciplined first draft; the final 10–20% of judgement, validation and polish is human, and that is by design.

3What it leverages behind the scenes

Five layers. The framework is just the middle one — the “plumbing” forked and configured. The intelligence is the AI assistant already in use; the outputs go to systems already owned.

▲  intent flows down  ·  artefacts flow back up  ▼
👤 1. You (the human)
State the intent — paste a transcript, type an idea, review & approve at each gate. You own the what and the why.
🧠 2. The “brain”
Cursor + LLM
Does the actual reading, reasoning and writing — transcript → stories → plan → tasks → tests → code — on Cursor + a large language model (Claude / GPT-class). The only AI in the stack.
⚙️ 3. The engine
(framework base, forked)
The SDD framework is forked into a Salesforce edition with the rules, templates, decisions and files engineered in — CLI, templates, scripts and /speckit-* prompts. The framework gives the method; the build makes it produce correct Salesforce artefacts. It holds no AI — it just hands the brain a consistent structure and the next prompt.
🏥 4. Project customisation
The Salesforce edition adds the Constitution, memory (taxonomy / domain / regulatory), domain-tuned templates, the test-case generator, the dashboard & token tracker, and Jira wiring.
📤 5. Where output lands
Files land in Git → deploy to the Salesforce org (sf CLI) → track in Jira → publish to a GitHub Pages dashboard. All systems already in use — no new data destination.
❌ Common misconception
“It’s a special AI plugin that converts transcripts into stories, plans and tests.”
✅ What actually happens
No separate plugin. At each step the engine hands the assistant a structured prompt + template, and the language model does the conversion. The engine’s value is the discipline, repeatability and guardrails around it.

4What it is built on — the engineering breakdown

Three kinds of pieces: the FRAMEWORK BASE that is forked, what is BUILT for Salesforce, and EXISTING TOOLS in the stack. The framework is the start; the rules, templates, governance and wiring that make it deliver are the added work.

ComponentTypeWhat it isRole in the flow
Spec-Driven-Development framework (the base that is forked) FRAMEWORK BASE The proven SDD method & CLI as a starting point, forked into the Salesforce edition. Provides the specify → plan → tasks → implement method then configured for Salesforce.
Cursor + LLM EXISTING TOOL The AI coding assistant and its language model (Claude / GPT-class). The brain. Reads the transcript and writes every artefact and the code.
Salesforce templates & commands BUILT Domain-tuned templates (spec, plan, tasks, test-cases) + the /speckit-* command prompts. Tell the AI exactly what shape each Salesforce output must take — consistency across people & features.
Installer CLI (speckit) BUILT One command that sets up a project with all the rules, templates & files (via uv/pipx/pip). One-time setup & updates across teams. Copies the engineered files in — never calls an AI.
Git + helper scripts (Bash / PowerShell) EXISTING TOOL Standard version control + the scripts that create feature folders, resolve paths and read JSON. Numbering features, wiring files together, keeping everything in source control.
Constitution & memory BUILT Salesforce governance rules + taxonomy / domain / regulatory context, grounded in the relevant Salesforce industry-cloud models. Guardrails checked at every gate; gives the AI project-specific context.
Progress dashboard + token tracker BUILT A dashboard (HTML/CSS/JS) reading a progress-tracker.json file. Live program status & effort transparency, published on GitHub Pages.
Jira & GitHub EXISTING TOOL Work-tracking and code-hosting platforms. Stories/status sync to Jira; repo + dashboard hosting on GitHub.
Salesforce sf CLI EXISTING TOOL Salesforce’s standard deployment tool. Pushes the generated metadata/code into the org at the Implement step.

5Why it matters — talking points

In plain terms: the AI is a brilliant, fast worker; the engine is the assembly line, jigs and quality checklist around it. The result is output that’s consistent, inspected at every station, traceable to the original intent, and reproducible by anyone — craftsmanship at factory discipline.
Intent is the asset. The spec (what & why) is captured once and reused — it doesn’t rot like a Word doc. Code is the renewable output.
Quality gates, not vibes. Ambiguity is removed and standards enforced before code — issues caught in design, not production.
Full traceability. Every task and test traces back to a user story; the dashboard shows live status and effort.
Engineered once, reusable everywhere. Rules, templates and governance on a forked framework — on tools already owned, no new data destination. The same engine powers the next project.
Repeatable across the team. Same commands, same structure — onboarding and handover become trivial.
Faster, safer delivery. A typical feature: one objective → spec → plan → traceable tasks → tests, with a security gap caught pre-build.

7Project grounding — the guardrails SpecKit reads every time

The framework is customer-agnostic, but every requirement is executed against project-specific memory & guardrails. These are loaded on every command (specify → plan → tasks → implement, plus clarify / wireframe / analyze) — enforced by the always-on rule .cursor/rules/grounding-guardrails.mdc (“Project Grounding & Guardrails — READ FIRST”). Nothing is written that isn’t traceable back to them.

GuardrailWhere it livesWhat it locks in
Constitution .specify/memory/constitution.md Non-negotiable architecture, security (USER_MODE, with sharing, FLS), governor limits, testing (≥75%), permission-set governance, Industry-First & Salesforce industry data model alignment
Project Details .specify/memory/project-details.md Org alias & API version, connected systems, modules, scope / out-of-scope, Jira & repo IDs, grounding index
Domain .specify/memory/domain.md Business processes & guiding principles for the target domain
Taxonomy .specify/memory/taxonomy.md Canonical naming & terminology — objects, records, roles (used verbatim)
Regulatory .specify/memory/regulatory.md Industry & data-governance compliance guardrails relevant to the engagement
Data Model force-app/…/objects REAL object/field API names from the org — inspected before any field list; fields are never invented
Wireframe Anatomy .cursor/rules/wireframe-salesforce-anatomy.mdc Mandatory Lightning record-page structure for every wireframe / Figma frame / screen mock
Always-On Rules .cursor/rules/grounding-guardrails.mdc The READ-FIRST mandate itself, spec-authoring depth, domain field grounding & Git/Jira/org/dashboard deploy-sync discipline
Templates .specify/templates/*.md Required section structure for spec / plan / tasks / checklist / test-cases
Precedence On any conflict: Constitution → Rules (grounding-guardrails.mdc) → Memory (domain / taxonomy / regulatory / project-details) → Templates. If a needed artifact is missing or still has unfilled placeholders, SpecKit flags it instead of assuming a default.
